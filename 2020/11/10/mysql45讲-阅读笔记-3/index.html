事务隔离：为什么你改了我还看不见？

事务隔离例子——银行转账：查询余额、做加减法、更新余额，操作必须是一体的，否则还没做减法之前，完全可以再查一次然后再给别人转账。

事务：简单来说就是保证一组数据库操作，要么全部成功，要么全部失败。

MySQL中，事务支持是在引擎层实现的，而MySQL是一个支持多引擎的系统，不是所有的引擎都支持事务，比如MySQL原生的MYISAM引擎不支持事务，而InnoDB引擎支持。

本文以InnoDB为例，剖析MySQL在事务支持方面的特定实现，并给予原理给出相应的实践建议，以此加深对MySQL事务原理的理解。

隔离性与隔离级别

事务的四个特性：ACID（原子性Atomicity、一致性Consistency、隔离性Isolation、持久性Durability）

SQL标准的事务隔离级别及其可避免的问题：

隔离级别 英文 脏读 dirty read 不可重复读 nonrepeatable read 幻读 phantom read
读未提交 read uncommitted 存在 存在 存在
读提交 read committed 不存在 存在 存在
可重复读 repeatable read 不存在 不存在 存在
串行化 serializable 不存在 不存在 不存在

首先，隔离的级别越高，效率越低，很多时候需要找到两者之间的平衡点（酸碱平衡理论，BASE碱原则）

  • 读未提交：一个事务还未提交，其变更就能被别的事务看到；

  • 读提交：一个事务提交之后，其变更才能被别的事务看到；

  • 可重复读：一个事务执行过程中看到的数据从事和启动时看到的数据一致；

  • 串行化：对于同一行记录，写操作会加写锁，读操作会加读锁。当出现读写锁冲突时，后访问的事务必须等前一个事务执行完成，才能继续执行。

不同隔离级别下V1，V2，V3的值：

  • “读未提交”，V1=V2=V3=2；
  • “读提交”，V1=1，V2=V3=2；
  • “可重复读”，V1=V2=1，V3=2；
  • “串行化”，V1=V2=1，V3=2。与“可重复读”不同的是，事务B在执行“将1改成2”时，会被锁住，直到事务A提交后，事务B才可以继续执行。

实现原理：

数据库会创建视图（MVCC，区分版本号，包含当前存在的事务版本号，寻找可用的记录），访问时以视图的逻辑结果为准：

  • ”可重复读“在事务启动时创建视图，在整个事务存在期间都使用该视图；
  • “读提交”在每个SQL语句开始执行时创建；
  • ”读未提交“直接返回记录上的最新值，没有视图概念；
  • “串行化”采用加锁的方式避免并行访问。

Oracle数据库的默认隔离级别为“读提交”，为了保证数据库的隔离级别一致，对于一些从Oracle迁移到MySQL的应用，需要将MySQL的隔离级别设置为“读提交”：

将transaction-isolation的值设置为READ-COMMITTED，可以使用show variables命令查看当前隔离级别。

”可重复读“的具体实现：

MySQL中，每条记录在更新的时候都会同时记录一条回滚操作。最新值通过回滚操作可以得到前一个状态的值。假设一个值初始为1，被按顺序改成了2、3、4，在回滚日志里面会有类似下面的记录。

为什么尽量不要用长事务？

长事务意味着系统里面会存在很老的任务视图，导致这个事务提交之前，所有可能用到的回滚记录都必须保留，占用大量的存储空间。

在MySQL 5.5及其之前的版本，回滚日志是和数据字典一起放在ibdata文件中的，即使长事务最终提交，回滚段被清理，文件也不会变小。为了减小存储空间占用，只能重建整个库。

长事务还会占用锁资源，可能会拖垮整个库。

事务的启动方式

1、显式启动事务语句，begin或者start transaction，对应的提交语句为commit，回滚语句是rollback。

2、set autocommit=0，这个命令会将线程的自动提交关掉。意味着只执行一个select语句，事务就启动了，并且不会自动提交，这个事务存在直到主动执行commit或rollback语句，或者断开连接。

有些客户端会默认在连接成功之后先执行一个set autocommit=0的命令，导致接下来的查询都在事务中，如果是长连接，就导致了意外的长事务。

建议使用set autocommit=1，通过显式语句的方式启动事务。

在autocommit为1时，使用begin显式启动事务。如果执行commit提交事务，而使用commit work and chain命令则是提交事务并自动启动下一个事务，省去再次执行begin语句的开销。从开发的角度，明确知道每个语句是否处于事务中。

小结

本文介绍了MySQL的事务隔离级别的现象和实现，分析长事务的风险，以及如何正确避免长事务。

如何避免长事务对业务的影响？

  • 应用开发端：

    1、确认autocommit为1，使用general_log来确认

    2、确认是否有不必要的只读事务，（事务中只有select语句）

    3、业务连接数据库的时候，根据业务本身的预估，通过SET MAX_EXECUTION_TIME命令，控制每个语句执行的最长时间，避免单个语句意外执行太长时间

  • 数据库端：

    1、监控information_schema.Innodb_trx表，设置长事务阈值，超过就报警或者kill

    2、Percona的pt-kill工具

    3、业务功能测试阶段要求输出所有的general_log，分析日志行为提前发现问题

    4、如果使用MySQL 5.6或更新的版本，将innodb_undo_tablespaces设置为2或更大。如果真的出现大事务导致回滚段过大，这样设置会使得清理更方便。
